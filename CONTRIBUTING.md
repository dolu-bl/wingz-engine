## Основные принципы

1. **Читаемость превыше всего.** Код пишется для людей, а не для машин. Он должен быть понятен вашему коллеге (и вам самому спустя полгода) без лишних усилий.
2. **Следуйте принципу наименьшего удивления.** Ваш код должен быть интуитивно понятен в контексте проекта.
3. **Консистентность — ключ к успеху.** Придерживайтесь стиля, уже существующего в файлах, которые вы редактируете. Новый код должен быть неотличим от окружающего.
4. **Компилятор — ваш друг.** Код не должен генерировать предупреждений (warnings) при сборке. Все предупреждения должны быть устранены.

## Форматирование и стиль кода

### Отступы и скобки

- **Отступы:** Используйте **только пробелы**. Размер отступа — **4 пробела**.
- **Стиль скобок (Allman/BSD):** Открывающая и закрывающая фигурные скобки располагаются на отдельных строках, на том же уровне отступа, что и управляющая конструкция. Это улучшает читаемость и визуально разделяет блоки кода.

```cpp
// Правильно
if (condition)
{
    doSomething();
}

class MyClass
{
public:
    MyClass();
};
```

- **Однострочные конструкции:** Разрешается опускать скобки для однострочных `if`, `for` и т.д., если это не ухудшает читаемость. Однако, если в конструкции есть `else` или она вложена, скобки обязательны.

```cpp
// Допустимо
if (!data)
    return;

// Обязательно
if (first_condition)
{
    if (second_condition)
        foo();
    else
        bar();
}
```

- **Длина строки:** Старайтесь не превышать **110 символов** в строке. Разделяйте длинные выражения на несколько строк с логическим выравниванием.

### Именование

В проекте используются следующие конвенции именования:

| Сущность                   | Стиль                       | Пример                           |
| :------------------------- | :-------------------------- | :------------------------------- |
| **Файлы**                  | lower_case_with_underscores | `sqlite_user_repository.cpp`     |
| **Пространства имён**      | lower_case_with_underscores | `namespace server::handlers`     |
| **Классы / Структуры**     | UpperCamelCase              | `AuthHandler`, `UserNotification`|
| **Интерфейсы**             | `I` + UpperCamelCase        | `IAuthMiddleware`, `IResultSet`  |
| **Методы / Функции**       | lowerCamelCase              | `handleLogin`, `extractItemId`   |
| **Члены класса**           | `m_` + lowerCamelCase       | `m_isRunning`, `m_blacklist`     |
| **Переменные**             | lowerCamelCase              | `itemId`, `userLogin`            |
| **Константы / Enum class** | UpperCamelCase              | `enum class MessageLevel`        |

- **Аксессоры:** Геттеры пишутся без префикса `get` (`int size() const`), а сеттеры — с префиксом `set` (`void setSize(int size)`).
- **Макросы:** UPPER_CASE_WITH_UNDERSCORES.

### Include-директивы

- Порядок включения заголовочных файлов должен быть сверху вниз от наиболее общих к наиболее специфичным:
    1. Стандартная библиотека C++ (`<string>`, `<vector>`)
    2. Внешние библиотеки (`<boost/...>`, `<nlohmann/json.hpp>`, `<cpprest/...>`)
    3. Заголовки проекта (`"common/dto/user.h"`, `"api/middleware/impl/auth_middleware.h"`)
- Группы должны быть разделены пустой строкой.
- Используйте **кавычки** (`""`) для собственных заголовков и **угловые скобки** (`<>`) для библиотечных.

### Организация объявлений внутри класса

Объявления внутри класса должны следовать строгому порядку, который ставит публичный интерфейс на первое место. Секции располагаются в следующей последовательности: `public`, затем `protected`, и в конце `private`. Отсутствующие секции просто пропускаются.

Внутри **каждой** секции придерживайтесь такого порядка:

1. **Псевдонимы типов** (`using`, `typedef`) и перечисления (`enum`, `enum class`)
2. **Константы и статические члены** (`static const`, `const`, `static`)
3. **Конструкторы, деструктор и методы класса**
4. **Поля класса**

Такой подход гарантирует, что читатель в первую очередь видит публичное API класса — именно то, с чем ему предстоит работать. Для визуального разделения логических групп внутри одной секции допустимо повторно указывать модификатор доступа.

```cpp
// Правильно: публичный интерфейс в начале, поля в конце
class GoldenHammer
{
public:
    using SizeType = int;

public:
    explicit GoldenHammer(SizeType size);
    virtual ~GoldenHammer();

    SizeType size() const;

protected:
    void attack();

private:
    void hit();

private:
    SizeType m_size;
};
```

## Правила написания кода

### Современный C++

Проект нацелен на использование возможностей современных стандартов C++.

- Используйте `auto` для сложных типов и итераторов, если это не вредит читаемости.
- Предпочитайте `enum class` старым `enum` для строгой типизации.
- Используйте `nullptr` вместо `NULL` или `0`.
- Для владения объектами используйте умные указатели (`std::unique_ptr`, `std::shared_ptr`). Избегайте сырых `new` и `delete`.
- Для приведения типов используйте операторы C++ (`static_cast`, `dynamic_cast`, `const_cast`), а не C-style cast.
- Активно используйте `std::optional` для возврата значений, которые могут отсутствовать (см. `SqliteUserRepository::findByLogin`).

### Комментарии и документация

- **Язык комментариев:** Русский (как было в предоставленных файлах). Весь код и документация должны быть на русском языке.
- **Комментируйте "зачем", а не "что".** Код сам по себе должен объяснять, *что* он делает. Комментарий должен объяснять неочевидные решения, *зачем* это было сделано.
- **Формат Doxygen:** Используйте `///` для однострочных и `/** ... */` для многострочных комментариев.
- **Документируйте интерфейсы:** Каждый публичный метод в заголовочном файле `.h` должен иметь краткое описание. Реализация в `.cpp` может содержать более детальный комментарий с описанием параметров.

```cpp
// .h файл
/// Находит пользователя по логину.
std::optional<dto::User> findByLogin(const std::string& login) override;

// .cpp файл
/**
 * @brief Ищет пользователя по логину в БД.
 * @param login Логин пользователя.
 * @return DTO пользователя или std::nullopt, если пользователь не найден.
 */
std::optional<dto::User> SqliteUserRepository::findByLogin(const std::string& login) { ... }
```

- **TODO-метки:** Оставляйте TODO-метки с пояснениями для мест, требующих доработки.

```cpp
// TODO: Здесь должна быть логика получения элементов из БД с учетом прав доступа.
```

### Работа с ошибками

- Используйте исключения (`std::runtime_error`, `std::invalid_argument`) для обработки критических и неожиданных ошибок, как это делается в слоях `storage` и `repo`.
- На уровне API (`handlers/`) перехватывайте исключения и возвращайте соответствующий HTTP-статус с понятным сообщением в формате JSON (см. `RestServer::handleRequest`).
- Не используйте возвращаемые коды ошибок там, где можно использовать исключения. Возврат `std::optional` хорош для ожидаемого отсутствия значения (например, объект не найден), а не для ошибки выполнения.

## Тестирование

Мы используем **Boost.Test** для модульного и интеграционного тестирования. Тесты находятся в директории `tests/`.

- **Структура тестов:** Отражает структуру исходного кода (`tests/dto/`, `tests/storage/`, `tests/repo/`).
- **Именование тестов:**
  - Тестовые сьюты: `BOOST_AUTO_TEST_SUITE(MyComponentTests)`
  - Тестовые случаи: `BOOST_AUTO_TEST_CASE(test_my_specific_case)`
- **Фикстуры:** Используйте фикстуры (`BOOST_FIXTURE_TEST_CASE`) для организации повторяющегося кода инициализации и очистки ресурсов.
- **Моки:** Для тестирования серверной логики без реальной БД или зависимостей используются моки (см. `tests/server_mocks/`). Создавайте интерфейсы и их мок-реализации для изоляции тестируемого компонента.
- **Покрытие:** Каждая новая функция или метод, особенно в слоях `logic`, `repo` и `api`, должны сопровождаться тестами.

## Процесс внесения изменений (Workflow)

1. **Создайте задачу (Issue).** Любое изменение должно начинаться с обсуждения в задаче на GitHub/GitLab.
2. **Создайте ветку.** Для работы над задачей создайте отдельную ветку от `main` (или `develop`) с говорящим названием, например: `feature/UC01-create-root-project`, `bugfix/auth-token-validation`.
3. **Пишите код, следуя этому гайду.**
4. **Покройте свой код тестами.** Убедитесь, что все тесты проходят локально.
5. **Убедитесь в отсутствии предупреждений.** Проверьте вывод компилятора при сборке.
6. **Создайте Pull Request.**
    - Опишите, *что* было сделано и *зачем*.
    - Свяжите PR с задачей (например, `Closes #123`).
    - Убедитесь, что CI-пайплайны (сборка и тесты) проходят успешно.
7. **Ревью кода.** Один или два мейнтейнера должны просмотреть ваш код. Будьте готовы к обсуждению и правкам.
